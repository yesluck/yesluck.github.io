ML Strategy (1)

发表于 2018-02-05 | 分类于 Deeplearning.ai Notes |

本文首发于本人知乎专栏。 (This passage was first published in Zhihu.)

第一周的标题便是“ML Strategy (1)”，即ML的技巧。众所周知，虽然我们大多数人都会ML，但是如果不掌握相应的技巧，ML的过程不仅不会带来快乐，还会变得痛苦。因此，我们就认真学习ML的技巧，多多ML吧！

本周笔记摘自“deeplearning.ai”第三门课程“Structuring Machine Learning Projects”的Week 1。

Introduction to ML Strategy

这课竟然还没有中文字幕，喷了……不过其实该课程很多内容都和吴恩达教授本人的“Machine Learning Yearning”一书有很大重合，都是他人生经验的汇总啊。

  • Orthogonalization（正交化）
    • 一种操纵只控制一个功能
    • 思想运用于机器学习
      • 训练集的cost function数值可接受（接近人类表现）——更大网络、更好优化器
      • dev集的cost function数值可接受——正则化、更多训练集
      • 测试集的cost function数值可接受——增大dev集
      • 现实生活中运行良好——改变dev集或cost function

Setting up your goal

  • Single number evaluation metric（单一数字评估指标）
    • 例如：查准率（Precision）、查全率（Recall）有时存在tradeoff
    • 解决：使用F1 Score（P与R的调和平均数），能直接分辨分类器优劣
    • dev集+Single number evaluation metric→加快找到最优模型
  • Satisficing and Optimizing metric（满足指标和优化指标）
    • 例如：准确率和运行时间的tradeoff
    • 解决：准确率作为“优化指标”，运行时间作为“满足指标”（不满足的不予考虑）
    • 再例：N个不同的标准，设置1个为“优化指标”，(N-1)个为“满足指标”
    • 再例：wake words / trigger words（唤醒关键词），设置准确率为“优化指标”，被假唤醒次数为“满足指标”
  • Train/dev/test sets（这其实课程二讲过）
    • dev集和测试集应该来自同一分布
    • dev集和测试集应该能反映我们未来预期得到的或者重要的数据
    • Train/dev/test比例分配应考虑总数据量
    • 测试集大到足以对系统整体性能进行评估，有时不一定需要
  • 何时改变dev/test sets和metrics
    • 问题：dev/test sets或metrics与目标(preference)不完全一致
    • dev/test sets+metrics好不代表最终应用效果就好，如果出现不一致则需更改dev/test sets或metrics

Comparing to human-level performance

  • 为什么要human-level performance
    • 贝叶斯偏差（Bayes optimal error）: the best possible error. 无法超越
    • human performance时期缓慢提高：
      • human performance经常接近Bayes optimal error
      • <时期人能标训练集
      • <时期能分析人的操作
      • <时期更好地分析bias和variance
  • 可回避的偏差（Avoidable bias）：贝叶斯偏差（或其估计）与训练误差之差
  • human-level performance：最优的人类表现结果，是proxy for Bayes error

  • 提高表现
    • 两个监督学习的基本假设
      • 能拟合好训练集（Avoidable bias）
      • dev/测试集也能有训练集表现（Variable）
    • 综合思路
